本课回答四个问题：Agent到底是什么？Skill怎么被调用？交易经验怎么写成程序？AI在哪里参与、又被限制在哪里？

开场先纠正一个常见误区：这不是“让大模型直接炒股”，而是一套以确定性计算为核心、AI研究为增强的本地投研系统。
01 · PROBLEM

为什么不能直接把Excel交给大模型

结果不可复现

同一份数据可能因为Prompt、上下文或模型变化产生不同信号。

正式状态容易混淆

AI可能把“密切观察”改写成“可以买入”。

数据边界不清楚

无法准确证明用了哪天、哪只标的、多少历史数据。

维护成本失控

每次调整ETF规则都需要同时改Prompt、页面和解释逻辑。

一个真实会遇到的问题

用户问：“黄金ETF为什么没有确认中期趋势？”如果直接把Excel交给模型，模型可能只看到当天MACD改善就给出偏乐观结论，却漏掉MA20仍向下或周线条件没有通过。

系统做法：先由代码产生正式状态和缺失条件，再由AI解释“哪些满足、哪些没满足”。

最终选择：代码计算正式结果，AI读取受控证据做研究解释。
这一页讲设计动机。不要先讲技术名词，先让听众理解为什么要做职责分离。

03 · CONCEPTS

Agent、Skill、Plugin、Tool不要混为一谈

Workflow Agent

带输入输出契约的代码任务节点，负责调度与治理。

Skill

一类业务能力的实现入口，由Handler执行。

Strategy Plugin

正式策略算法，实现统一输入输出接口。

Research Tool

投研问答向AI开放的只读数据能力。

Workflow Agent
→
Skill Handler
→
Strategy Plugin
→
Decision
关键区别：投研问答不会直接读取ETF的SKILL.md来计算策略，它走独立的Tool Registry。
04 · DAILY WORKFLOW

十四个Agent，本质是确定性代码流水线

阶段Agent职责
输入准备Config / SourceArchive / Data / Portfolio读取配置、归档源文件、解析Excel、读取持仓
数据处理QA / Indicator质量检查，统一计算技术指标
资产策略ETF / TL / ConvertibleBond调用各自确定性策略模块
研究风控Backtest / Risk / AICommittee / Explanation历史诊断、风险汇总、可选文字复核
输出Report生成日报、JSON和前端数据
当前实际状态：每日刷新中的LLM文字复核默认关闭，因此策略、评分、排名和报告均可离线运行。
一次刷新是怎样向后传递的

DataAgent读取ETF原始表 → IndicatorAgent增加MA5、MA10、MA20和MACD → ETFAgent产生状态 → ReportAgent只负责把结果写入Dashboard。

任何后续节点都不重新猜测前一个节点已经确定的结果。

AIResearchCommitteeAgent和ExplanationAgent虽然预留LLM，但daily_report_llm_enabled=false时走确定性模板。投研问答的LLM开关与此分离。
05 · ETF AGENT

ETFAgent不是策略，只是ETF模块入口

class ETFAgent(SkillBackedAgent):
  name = "etf-agent"
  spec = AgentSpec(
    role="ETF趋势筛选Agent",
    skill_name="etf-rotation-strategy",
    required_artifacts=(
      "etf_indicators",
      "positions",
      "strategy_params"
    ),
    produced_artifacts=(
      "etf_signal_table",
      "etf_buy_candidates",
      "etf_watchlist",
      "etf_sell_alerts"
    )
  )

role/objective
审计与职责说明，不会被AI解释执行。

skill_name
真正的调用入口。

required_artifacts
运行前必须存在的数据。

produced_artifacts
运行后必须交付的结果。

ETFAgent负责“组织和验收”，策略插件负责“计算和判断”。
运行时的具体例子

运行前，Context里已经有：etf_indicators、positions、strategy_params。

运行后，必须出现：etf_signal_table、etf_buy_candidates、etf_watchlist、etf_sell_alerts。

少一个输出，Agent运行时就判定该节点失败，而不是让页面拿半套数据继续展示。

06 · SKILL

SKILL.md到底用来干什么

SKILL.md

记录Skill名称、用途、使用条件、输入输出和规则边界，属于能力契约与文档。

handler.py

真正被工作流运行时调用的Python实现，读取Context并写回结果。

ETFAgent
→
SkillRegistry.run()
→
handler.py
→
策略插件
问题准确答案
SKILL.md是Prompt吗？不是。正式策略不会把SKILL.md交给LLM理解。
谁执行Skill？日度工作流中的Skill Registry调用handler.py。
投研问答会用它吗？不会直接调用。问答使用独立的Tool Registry和结构化证据。
中文说明会决定信号吗？不会。信号由Python布尔条件、参数和状态机决定。
两个容易混淆的调用

每天刷新：ETFAgent → SkillRegistry → handler.py → ETF插件，会真正计算正式ETF状态。

投研问答：Router → ToolRegistry → etf_single_asset工具 → 结构化证据 → LLM，通常读取已有结果，不拿SKILL.md当Prompt。

SKILL.md负责说明能力和边界；handler.py与策略代码负责执行。

08 · RULES AS CODE

策略不是中文Prompt，而是公式、布尔条件和状态机

1. 指标与公式

MA、MACD、量能倍数、斜率、涨幅、阳线实体。

2. 布尔条件

大于、小于、上穿、绿柱缩短、是否全部满足。

3. 状态机

观察、确认、等待、过热、可入场、结构失效。

例：密切观察

ma_ready = ma5 > ma10
green_narrowing = macd_hist < 0 and macd_hist > previous_macd_hist
green_to_red = previous_macd_hist <= 0 and macd_hist > 0

close_watch = ma_ready and (green_narrowing or green_to_red)

例：中期确认

trend_confirmed = all([
  close > ma20,
  ma5 > ma20,
  ma20_state in {"flat", "up"},
  weekly_macd_hist > 0,
  daily_macd_hist > 0
])
数值例1：进入密切观察

MA5=0.864，MA10=0.854；昨日MACD柱=-0.006，今日=-0.002。

MA5已高于MA10，MACD仍是绿柱但绝对值缩小。

输出：close_watch。这不是趋势确认，更不是建仓候选。

数值例2：仍不能确认趋势

收盘=0.91，MA20=0.88，MA5也在MA20上方，但周MACD柱=-0.012。

价格条件满足，周线尚未转红。

输出：trend_not_confirmed，缺失项为周MACD确认。

09 · STATE MACHINE

ETF 2.0不是一次判断，而是一段状态变化

中期未确认→密切观察→趋势确认→等待突破
等待突破→突破确认 / 可入场
等待突破→等待回踩→回踩确认 / 可入场
任何短期阶段→过热不追→冷却观察
回踩→结构破坏 / 取消入场
过热保护

日涨幅≥4%、阳线实体≥60%、量能≥1.8倍、偏离MA5≥3%同时成立。

回踩确认

触及支撑、盘中结构未破、收盘站回、成交量收缩、日MACD红柱。

过热案例

当日涨5.2%，阳线实体65%，量能2.1倍，高于MA5 3.4%。

四项同时达到阈值：过热不追。

突破案例

确认日高点1.020；后续收盘1.031，MACD红柱，高于MA5仅1.8%。

突破路线确认：可以入场。

回踩案例

支撑1.000；最低0.997，收盘1.005，成交量低于确认日，MACD仍红。

缩量回踩并收回：可以入场。

状态机是本策略比简单buy/neutral更接近实际交易流程的地方。强调“趋势确认不等于立即建仓”。
10 · CONTRACT

插件真正稳定的是ETFDecision契约

身份与版本

日期、代码、名称、strategy_id、version。

正式状态

medium_status、short_entry_status、exit_status。

动作标记

buy_candidate、watch_candidate、sell_alert。

证据

rule_hits、missing_conditions、metrics。

风险与质量

risk_notes、confidence、data_quality。

兼容输出

供旧页面、日报和数据库统一消费。

前端只认识Decision，不需要知道策略内部使用了几条均线。
11 · OTHER ASSETS

主决策层与辅助观察层分离

TL

日/周MACD与KDJ决定主状态；30年国债ETF份额变化只提示资金方向和潜在拐点，不修改主状态。

可转债

legacy_v1决定排除、评分、资格与排名；dynamic_v2四项因素只解释短期联动，不修改基础排名。

稳定基础规则
+
新辅助证据
→
先解释、不改排名
→
验证后再升级
TL份额案例

TL技术规则输出“不做交易”，但30年国债ETF近5日份额合计增加0.08亿份。

页面提示“逆势流入，观察潜在转向”；TL正式状态仍然是不做交易。

可转债联动案例

正股-1.68%，转债-0.72%，相对表现+0.96个百分点，溢价率增加0.98个百分点。

动态层解释转债抗跌及溢价变化；基础分和最终排名保持不变。

架构原则：新因素先做Overlay，避免未经验证的数据直接改变正式决策。

同一个系统中的三种回答

“黄金ETF收盘价？” 规则工具直接返回数值，不调用LLM。

“黄金ETF中期趋势怎么样？” 先读取单标的状态、MA20、日周MACD，再由快速模型组织分析。

“ETF、TL和可转债是否共同反映风险偏好转弱？” 主Agent分步调用多个工具，检查证据后综合回答。

与日度Skill的关系：问答不直接运行ETF策略插件，而是通过工具读取已经入库的状态、指标和受控历史；双策略对照工具才会按明确契约重新计算。
13 · TOOLS & POLICY

给AI研究自由度，但不给系统控制权

AI可以
  • 查询单只ETF最近30日
  • 比较2–10只ETF当前状态
  • 计算最多400日双策略对照
  • 分析TL最近30日
  • 比较可转债前30只
  • 查询数据质量和风险摘要
AI不可以
  • 执行任意SQL或读取任意文件
  • 修改策略、参数、评分和排名
  • 刷新数据或写入数据库
  • 访问未接入的实时行情和新闻
  • 连接账户或自动下单
  • 把辅助状态改写成正式信号
14 · GOVERNANCE

证据审查在前，输出保护在后

EvidenceAccuracyReviewer

检查标的、日期、数量边界、重复记录、证据完整性，以及多标的是否全部读取。

OutputGuardrail

禁止把观察写成建仓、把辅助写成正式信号、修改评分排名或承诺收益。

证据审查通过，不代表投资结论一定正确；它代表本轮回答使用的数据满足系统契约。
15 · RELIABILITY

为什么刷新失败不会污染正式结果

读取与计算
→
Staging临时结果
→
报告与审计
→
SQLite入库
→
最后切换Dashboard
全部成功

一次性切换到新数据、新报告和新页面。

任一步失败

继续保留上一版成功结果，并显示失败阶段和处理建议。

失败案例：TL文件列名被改动

DataAgent可以明确定位TL读取失败；后续TL策略和审计不会把不完整结果发布到正式Dashboard。

ETF和可转债的上一版成功结果继续保留，数据状态页显示失败阶段和处理建议。

用户看到的是“上一版完整结果 + 本次失败说明”，而不是一张更新到一半的页面。

页面重载不会取消后台刷新任务；前端永远不负责计算正式策略。
16 · LESSONS

这套系统真正值得复用的设计

  1. Agent不是越智能越好：生产流水线更需要确定性和可审计。
  2. 策略不能依赖自然语言：交易经验要拆成公式、阈值与状态机。
  3. 插件稳定的是接口：内部算法可变，Decision契约不变。
  4. 新因子先做辅助层：经过验证后再考虑进入正式决策。
  5. AI只能通过工具取证：研究自由度与系统权限分开。
  6. 失败必须可回退：原子发布比“刷新成功率”更重要。
当前待治理项：ETFAgent中部分中文quality_gates仍接近原策略，应改成插件无关契约；不影响当前计算，但会影响架构说明的一致性。
17 · SUMMARY

最后记住三套系统

Workflow Agent

代码流水线，负责调度、治理、检查输入输出。

Strategy Plugin

确定性算法，负责正式信号、状态、评分与证据。

Research Agent

LLM+只读工具，负责查证、比较、解释与综合研究。

工作流节点把事情按顺序做完；策略插件把交易经验变成可复现代码；投研Agent只在需要语言理解和跨工具分析时出现。

这套架构的重点不是“AI自动交易”，而是让每个数据、状态和解释都能找到来源。

收尾时回到开场：系统价值不在Agent数量，而在职责边界、可复现结果和持续迭代能力。
