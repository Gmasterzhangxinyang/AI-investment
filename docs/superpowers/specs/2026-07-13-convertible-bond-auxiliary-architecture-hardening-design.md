# 可转债动态辅助与架构治理设计

## 目标

在不宣称原策略或动态策略已经被回测证明有效的前提下，把可转债模块改成清晰、可解释、可替换的结构：基础策略负责硬风控、资格、基础分和最终排名；动态策略只提供收盘后的短期辅助提示，不改变资格、动作或排名。同时补齐配置身份、全量历史、策略说明书和关键工程治理，避免页面出现用户不知道用途的字段。

## 已选方案

采用分层方案，不重写现有可转债模型：

1. `legacy_v1` 保留为基础决策插件，输出硬风控、基础分、基础等级、资格、动作和最终排名。
2. `dynamic_v2` 改为动态辅助插件，读取正股涨跌、转债涨跌和溢价率变化，只输出辅助分、辅助状态和短说明。
3. 动态辅助结果不进入基础分，不产生综合分，不改变资格、动作和排名。
4. 页面默认启用动态辅助插件，以便同时看到基础决策和有意义的短期提醒；关闭后仍可完整使用基础策略。

保留原策略是为了提供稳定、可复现的基准，不表示原策略已被证明更优。动态辅助暂不参与排名，是因为当前没有足够历史证据证明它能稳定改善收益或回撤。

## 用户可见语义

可转债主表保留：

- 名称、代码、行业、价格等必要信息。
- 基础分。
- 基础等级。
- 资格。
- 风险提示。
- 最终排名。

动态辅助只在以下状态出现明确文字：

- `关注补涨`：正股偏强、转债跟涨不足且溢价收缩。
- `谨慎追涨`：转债涨幅领先正股且溢价扩大。
- `联动走弱`：正股和转债同步明显走弱。
- `数据不足`：动态字段缺失或无法核验。

正常联动不占用主表列，不显示“正常联动”或空白的专业字段。详情区可以显示动态辅助分和原始分项，用于解释，不作为最终评分。

删除或隐藏以下误导性表达：

- 综合分。
- 动态等级。
- “综合分达到或未达到资格线”。
- 正常情况下为空的联动提示列。

所有资格和未入选原因必须明确以基础分和硬风控为准。例如基础分低于 30 时，显示“基础分低于30”，不能只写“评分低于30”。

## 插件架构

可转债模块使用与 ETF 相同风格的注册机制，但基础决策插件和辅助插件职责分开。

### 基础决策插件

稳定接口负责：

- 接收标准化可转债快照和基础策略参数。
- 返回全部标的的排除结果或候选结果。
- 对候选输出基础分、基础等级、资格、动作、风险和排名。

首个实现为 `legacy_v1`。未来新增基础策略时，通过注册表安装，不在总处理器中增加策略 ID 判断。

### 动态辅助插件

稳定接口负责：

- 接收已经完成基础决策的单行结果和标准化动态字段。
- 返回 `auxiliary_score`、`auxiliary_state`、`auxiliary_note`、`data_quality` 和分项解释。
- 不允许修改基础分、资格、动作、排名或排除结果。

首个实现为 `dynamic_v2`。总处理器只从注册表选择插件，不直接判断 `dynamic_v2`。

### 配置身份

每次结果必须记录：

- 基础策略 ID 和版本。
- 辅助策略 ID、版本和启用状态。
- 可转债完整配置哈希。
- 生成时间和源数据日期。

前端只有在策略 ID、版本和配置哈希全部匹配时，才显示“当前结果已按此配置生成”。只保存配置未刷新时显示“配置已保存，等待刷新”。

未知插件 ID 必须拒绝保存或明确失败，不允许静默切换。辅助插件单行数据异常时只把该行标为数据不足；辅助插件整体异常时关闭本次辅助结果并记录失败原因，基础决策仍可发布。

## 数据与历史保存

数据库按日报和债券代码保存当天全量可转债快照，包括：

- 进入候选池的标的。
- 被硬风控排除的标的。
- 基础策略输出。
- 动态辅助输出。
- 源数据实际日期。
- 策略身份和配置哈希。

日报日期与可转债源数据日期分开保存，不能用整个日报的最大日期替代可转债实际日期。数据库迁移保持向后兼容，旧记录缺少新字段时按空值读取。

这次只建立可追溯历史，不增加正式回测、不根据历史结果自动调参。

## 页面设计

主表只展示能直接帮助判断的字段，长原因和分项放到点击展开的详情区。表格必须设置合理最小宽度和横向滚动，禁止把长中文压成一字一行。

策略设置区使用两个概念：

- 基础策略：当前为原策略。
- 动态辅助：启用或关闭动态 v2。

不再把动态辅助包装成一个与基础策略竞争的“综合策略”。页面提供两份简短说明书入口，分别解释基础策略做什么、动态辅助做什么、哪些内容不会被动态辅助改变。

## 工程治理

本轮同时完成与该功能直接相关的治理：

- 拆出可转债基础策略注册表、辅助策略注册表、结果契约和配置身份模块，降低当前大处理器的职责。
- API Key 不再从页面写入仓库配置文件；页面只显示环境变量是否已配置。为兼容旧配置，读取时仍可识别旧字段，但保存时移除明文 Key。
- 增加依赖锁定文件，确保本地安装版本可复现。
- 增加真实浏览器端到端测试，覆盖保存辅助配置、提示等待刷新、刷新后身份匹配和主表字段显示。

不在本轮大规模重写整个前端或服务器；只拆分此次修改涉及的可转债和密钥逻辑。现有 URL、Excel 报告和确定性 AI 问答保持兼容。

## 不在本轮修改

- Wind/Excel 手工更新后再点页面刷新的工作流。
- ETF 原策略与 ETF 2.0 规则。
- TL 技术规则和份额变化规则。
- 正式组合回测、自动调参或收益承诺。
- 自动下单或券商连接。

TL 份额变化按收盘后数据处理，只能用于收盘后复盘和下一交易日参考。

## 测试与验收

实施遵循测试先行，至少覆盖：

1. 基础策略在启用和关闭动态辅助时，排除结果、基础分、基础等级、资格、动作和排名逐项一致。
2. 动态辅助只输出约定字段，无法修改基础结果。
3. 页面不再出现综合分、动态等级或正常联动噪声。
4. “基础分低于30/50”等原因与基础分一致。
5. 配置权重或阈值变化会改变配置哈希，并使页面进入等待刷新状态。
6. 全量可转债均进入数据库，排除标的保留排除原因和实际数据日期。
7. 旧数据库和旧 dashboard 可以读取，缺少辅助字段时安全降级。
8. 页面在常用桌面宽度下不出现一字一行的长文本。
9. 模型 Key 不会通过配置保存接口写回明文。
10. Python 单元测试、前端单元测试、数据库迁移测试和浏览器端到端测试全部通过。

## 成功标准

用户看到一只可转债时，可以直接回答：基础模型为什么这样排名、它是否有硬风险、动态辅助是否出现值得注意的短期异常。用户不会把辅助分理解为新的交易资格，也不会因为修改配置但尚未刷新而看到错误的“已经生效”状态。
